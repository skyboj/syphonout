# Virtual Display System Test Plan

## Тест-окружение
- macOS: 12.0+ (Monterey)
- SyphonOut: последний коммит (2785578)
- OBS: с obs-solink plugin (опционально для тестов с видео)

## Группа 1: Virtual Display базовые операции

### Test VD-1.1: Создание нового Virtual Display
**Описание:** Проверка создания нового VD через меню

**Steps:**
1. Запустить SyphonOut
2. Открыть меню в статус-баре
3. Выбрать "New Virtual Display…"
4. Проверить появление нового VD в списке

**Expected:**
- Новый VD с именем "Display N" (где N = текущее количество + 1)
- UUID создан в Rust core (`syphonout_vd_create` вызван)
- VD появляется в `VirtualDisplayManager.shared.displays`
- Сохранение в UserDefaults (`virtualDisplays` ключ)

**Проверки:**
-Log `VirtualDisplayManager.shared.displays.count` увеличился на 1
- Проверить лог Rust core (вызов `vd_create`)
- Проверить UserDefaults данные

### Test VD-1.2: Переименование VD
**Описание:** Проверка изменения имени существующего VD

**Steps:**
1. Создать VD (если нет существующих)
2. Вызвать `VirtualDisplayManager.shared.setName(vdId: "имя")`
3. Проверить обновление имени в меню

**Expected:**
- Имя обновляется в UI меню
- Rust core получает вызов `syphonout_vd_set_name`
- UserDefaults обновляется

**Проверки:**
- Имя в меню соответствует установленному
- Проверить лог Rust core
- Проверить UserDefaults после перезапуска приложения

### Test VD-1.3: Изменение разрешения VD
**Описание:** Проверка изменения разрешения VD через меню

**Steps:**
1. Выбрать VD в меню
2. Открыть подменю "Resolution"
3. Выбрать разрешение (например, 1280×720)
4. Проверить обновление

**Expected:**
- Разрешение обновляется в Rust core (`syphonout_vd_set_size`)
- UI отображает новое разрешение
- UserDefaults сохраняет новые размеры

**Проверки:**
- Разрешение в меню обновляется
- Rust core лог вызова `vd_set_size`
- Сохранение после перезапуска

### Test VD-1.4: Удаление VD
**Описание:** Проверка удаления VD через меню

**Steps:**
1. Создать минимум 2 VD
2. Для второго VD выбрать "Delete" в меню
3. Проверить удаление

**Expected:**
-TД удаляется из списка в меню
- Rust core получает `syphonout_vd_destroy`
- Очистка связанных ресурсов (IOSurface если был)
- UserDefaults обновляется

**Проверки:**
-cVD исчезает из меню
- `VirtualDisplayManager.shared.displays.count` уменьшается на 1
- Проверить отсутствие утечек памяти

## Группа 2: Preview система (миниатюры)

### Test PREV-2.1: Миниатюры для всех режимов
**Описание:** Проверка генерации миниатюр для каждого режима VD

**Steps:**
1. Создать VD
2. Проверить миниатюры для каждого режима:
   - Signal (требуется активный источник)
   - Freeze (требуется предварительный кадр)
   - Blank Black
   - Blank White
   - Test Pattern
   - Off

**Expected:**
- Signal: миниатюра из IOSurface через `PreviewRenderer.iosurfaceThumbnail()`
- Freeze: миниатюра из замороженного кадра
-

- Blank Black: solid черный квадрат
- Blank White: solid белый квадрат
- Test Pattern: SMPTE bars
. Off: отсутствие миниатюры (nil)

**Проверки:**
-X Визуальная проверка миниатюр в меню
- Для Signal/Freeze: проверка вызова `syphonout_vd_get_iosurface()`
- CFRetain/CFRelease баланс (отсутствие утечек)

### Test PREV-2.2: Производительность генерации миниатюр
**Описание:** Проверка скорости и ресурсов при генерации миниатюр

**Steps:**
1. Создать VD в Signal режиме с активным источником
2. Измерить время генерации миниатюры
3. Проверить memory usage
4. Проверить GPU нагрузку

**Expected:**
- Время генерации < 16ms (для 60fps обновления)
- Отсутствие утечек IOSurface
- Низкая GPU нагрузка (< 5% для downscale операции)

**Проверки:**
- Instruments: Memory Leaks, CPU Profiler
. Metal System Trace для GPU
  
### Test PREV-2.3: Интеграция миниатюр с меню
**Описание:** Проверка отображения и обновления миниатюр в меню

**Steps:**
1. Создать несколько VD с разными режимами
2. Проверить отображение миниатюр в меню
3. Сменить режимы, проверить обновление миниатюр
4. Получить новые кадры (для Signal режима), проверить обновление

**Expected:**
. Миниатюры отображаются над именем каждого VD
- Обновление при смене режимов
- Обновление при получении новых кадров
- Правильное позиционирование и размер (160×90)

**Проверки:**
- Визуальная проверка в меню
- Логи `PreviewRenderer.thumbnail()` вызовов
- Отсутствие артефактов или некорректных изображений

## Группа 4: Интеграция с UI и hotkeys

### Test UI-4.1: Меню управление
**Описание:** Проверка динамического построения меню

**Steps:**
1. Создать несколько VD с разными состояниями
2. Открыть меню, проверить структуру
3. Изменить состояния VD, проверить обновление меню

**Expected:**
-(Динамическое построение через `MenuBuilder.build()`
- Корректное отражение состояния каждого VD
- Реакция на изменения (подключение/отключение дисплеев)

**Проверки:**
- Структура меню соответствует `MenuBuilder` логике
- Состояния radio buttons корректные
-lОбновление при изменениях

### Test UI-4.2: Глобальные hotkeys
**Описание:** Проверка работы глобальных hotkeys

**Steps:**
1. Создать несколько VD
2. Проверить каждый hotkey:
   - ⌃⌥F: Freeze all outputs
   - ⌃⌥U: Unfreeze all outputs
   - ⌃⌥B: Blank all outputs (black)
   - ⌃⌥S: Restore all outputs to Signal

**Expected:**
- Hotkeys работают через NSEvent monitor
- Все VD реагируют на batch операции
- UI обновляется после hotkey операций

**Проверки:**
- `StatusBarController.registerGlobalShortcuts()` логика
-SРеакция всех VD
- Лог операций в Rust core

### Test UI-4.3: Batch операции
**Описание:** Проверка массовых операций над всеми VD

**Steps:**
1. Создать 3+ VD с разными состояниями
2. Выполнить каждую batch операцию:
   - Freeze all
   - Unfreeze all
   - Blank all
   - Restore all

**Expected:**
- Все VD переходят в соответствующий режим
- UI меню обновляется
-HoUserDefaults сохраняет состояния

**Проверки:**
- Все VD в одинаковом состоянии после операции
- Лог вызовов `VirtualDisplayManager.shared.setMode()`
1 Сохранение в UserDefaults

## Группа 5: Персистентность и состояние

### Test PERS-5.1: Сохранение состояния
**Описание:** Проверка сохранения и восстановления состояния VD

**Steps:**
1. Создать несколько VD с разными настройками:
   - Разные имена
   - Разные разрешения
   - Разные режимы
   - Разные источники (если есть)
2. Завершить SyphonOut
3. Перезапустить SyphonOut
4. Проверить восстановление состояния

**Expected:**
- Все VD восстанавливаются с теми же параметрами
- Имена, разрешения, режимы сохраняются
- Источники сохраняются (UUID)

**Проверки:**
- `VirtualDisplayManager.reconnectAll()` логика
- UserDefaults данные до/после
- Rust core инициализация VD при запуске

### Test PERS-5.2: Миграция с legacy состояния
**Описание:** Проверка совместимости с существующими данными

**Steps:**
1. Имитировать legacy UserDefaults данные
2. Запустить SyphonOut
3. Проверить обработку legacy данных

**Expected:**
1 Авто-создание default VD при первом запуске
- Обработка legacy форматов без crashes
-PegСовместимость с существующими пользовательскими данными

**Проверки:**
- `VirtualDisplayManager.load()` обработка ошибок
- Отсутствие crashes при поврежденных данных
- Graceful degradation

### Test PERS-5.3: UserDefaults корректность
**Описание:** Проверка структуры сохранения данных

**Steps:**
1. Создать сложную конфигурацию VD
2. Проверить структуру UserDefaults

**Expected:**
-, Структура сохранения `[VirtualDisplay]` в `virtualDisplays` ключе
- Структура сохранения `[CGDirectDisplayID: String]` в `vdAssignments`
- JSON кодирование/декодирование работает

**Проверки:**
- UserDefaults raw данные
- JSON валидность
- Обработка ошибок кодирования/декодирования
